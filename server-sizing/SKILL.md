# 服务器资源评估

## 适用场景

- 新客户签约后，部署神策系统前的资源规划
- 现有客户数据量增长，需要扩容评估
- 客户反馈系统性能下降，需要判断是否资源不足

## 核心原则（Iron Law）

**必须基于客户实际数据量和增长预期评估，不允许直接套用默认配置。**

默认配置只适合 POC 环境，生产环境必须按实际规模计算。

## 执行阶段

### Phase 1：数据规模收集

收集以下指标（现有客户从系统获取，新客户通过问卷收集）：

| 指标 | 说明 |
|------|------|
| 日均事件量 | 每天上报的事件总条数 |
| 峰值 QPS | 最高并发上报速率 |
| 用户规模 | 注册用户数 / 月活用户数 |
| 数据保留周期 | 需要保留多少个月的原始数据 |
| 查询并发数 | 同时使用分析系统的用户数 |

### Phase 2：增长预期确认

- 未来 12 个月的业务增长预期（用户数、事件量）
- 是否有大促/活动导致的峰值（倍数）
- 是否计划接入新的数据源

### Phase 3：资源计算

**存储估算：**
- 原始数据：日均事件量 × 平均事件大小(~500B) × 保留天数 × 3（副本）
- 索引数据：原始数据 × 0.3
- 总存储 = (原始 + 索引) × 1.2（预留空间）

**计算资源：**
- 写入节点：按峰值 QPS / 单节点处理能力(~5000 QPS) 计算节点数
- 查询节点：按并发查询数 / 单节点并发能力(~20) 计算节点数

**内存：**
- 每个节点建议内存 = 热数据量（最近 30 天）/ 节点数 × 0.3

### Phase 4：方案输出

输出三档配置：
- **推荐配置**：满足当前规模 + 12 个月增长 + 峰值余量
- **最低配置**：仅满足当前规模，无增长余量
- **扩容触发条件**：达到什么指标时需要扩容

## 常见问题

**客户无法提供准确数据量：** 要求客户提供 App/Web 的 DAU，按每用户每天 50-200 个事件估算。

**峰值 QPS 不清楚：** 按日均事件量 / 86400 × 10（峰值系数）估算。

**客户要求最低配置：** 明确告知最低配置的风险（无增长余量，扩容需要停机），让客户书面确认。
