# 服务器资源评估（CDP + MA）

## 背景知识

**产品体系：**
- **CDP**（客户数据平台）= SDH（SensorsData Horizon）+ SA（SensorsData Analysis），独立部署
- **MA**（营销自动化）= SF（Sensors Focus），独立部署，不与 CDP 共享节点

**CDP 部署模式：**

| 模式 | 内部名称 | 节点架构 | 特点 |
|------|---------|---------|------|
| 单机 | simplified_cluster | 1 节点，所有角色混部 | 适合小规模/POC |
| Mini 集群 | mini_cluster | 3 节点，所有角色混部（Hybrid） | 中等规模，最多扩至 5 节点 |
| 标准集群 | standard_cluster | 3 元数据节点 + N 数据节点（N≥3） | 大规模，支持横向扩容 |

**Mini 集群扩容限制：** 最多增加 2 个节点（共 5 节点），超过后需迁移至标准集群架构（有数据迁移成本），因此预期增长较快的客户建议直接上标准集群。

**架构要求：**
- 标准：仅支持 X86 架构
- ARM 架构：支持，但所有配置容量需 **double**（特殊情况单独处理）

## 执行阶段

### Phase 1：信息收集

向客户收集以下信息（已有 PROJECT.md 的先读取，避免重复询问）：

| 信息项 | 说明 | 如何获取 |
|--------|------|---------|
| 日均事件量 | 每天上报的事件总条数 | 客户提供或按 DAU × 每用户事件数估算 |
| 平均日活（DAU） | 日活跃用户数 | 客户提供 |
| 历史总事件量 | 已有的存量数据 | 客户提供，无则填 0 |
| 数据保留天数 | 需要保留多少天原始数据 | 通常 365 天，合规要求可能更长 |
| 未来 12 个月增长预期 | 事件量增长倍数 | 客户业务预测 |
| 服务器类型 | 云主机 / 物理机 | 影响磁盘扩容灵活性 |
| CPU 架构 | X86 / ARM | ARM 需要 double 配置 |
| 云厂商 | 阿里云 / 腾讯云 / AWS 等 | 影响磁盘类型选择 |
| 是否需要 SF（MA） | 是否部署营销自动化 | 独立评估 |

**DAU 估算事件量（无法提供时使用）：**
- 普通 App：DAU × 50～100 事件/天
- 电商/金融类 App：DAU × 100～200 事件/天

### Phase 2：CDP 部署模式选择

**决策树：**

```
日事件量 + 未来增长预期
│
├── ≤1500万/天（总量 ≤75亿）
│   └── 单机
│       ├── ≤1000万/天 → 标配（8C/64G）
│       └── ≤1500万/天 → 高配（16C/128G）
│
├── ≤1.5亿/天（总量 ≤1400亿）
│   └── Mini 集群
│       ├── ≤6500万/天 → 标配（8C/64G × 3）
│       ├── ≤1.5亿/天  → 高配（16C/128G × 3）
│       └── ⚠️ 若预期未来超过 3亿/天 → 建议直接上标准集群，避免迁移
│
└── >1.5亿/天 或 预期快速增长
    └── 标准集群（3 元数据 + N 数据节点）
        ├── ≤4.2亿/天  → 3+3 标配
        ├── ≤8.4亿/天  → 3+3 高配
        └── 更大规模   → 增加数据节点，顺序盘容量按计算逻辑单独估算
```

**优先推荐标准集群的场景：**
- 生产环境（支持横向扩容，无需数据迁移）
- 客户预算充足
- 未来 12 个月事件量增长超过 2 倍

**附加资源说明（在基础配置上叠加）：**

1. **用户关联策略**：参与用户关联的业务 ID 个数，每多 5 个则配置升一档
2. **动态可变关联策略**：若选择「动态可变关联策略」，配置升一档；目前不推荐这个配置
3. **数据 ETL 任务**：建议独立部署
4. **SF（MA）**：最初评估 SA 资源时，CPU 增加 ≥4C，Memory 增加 ≥8GB；且用户分群标签也会增加顺序盘空间；【SF 圈选用户，计算受众开销主要与运行中的计划数量也相关】
5. **Abtest**：最初评估 SA 资源时，CPU 增加 ≥4C，Memory 增加 ≥8GB；【选取受众的条件，计算受众的开销较大，跟运行中的试验数量也相关】
6. **SaaS SAT**：最初评估 SA 资源时，CPU 增加 ≥2C，Memory 增加 ≥4GB；【会通过 infinity client 通过 Impala 读写 Kudu 表】
7. **报表**：最初评估 SA 资源时，CPU 增加 ≥8C，Memory 增加 ≥16GB；【资源开销主要取决于业务市中创建业务模型选择的方法，还有查询的复杂度、数据量级、查询的时间周期、更新类型；还有配置的业务集市和报表的数量】
8. **数据导入**（涉及以下任意一种都会增加资源）：最初评估 SA 资源时，CPU 增加 ≥2C，Memory 增加 ≥2.5GB；如果涉及多中导入数据，需要累加计算。【Hive、MySQL、Oracle、FTP、微信公众号、微信小程序、搜索引擎/百度付费搜索、Kafka 数据流导入】
9. **JDBC 导出**：最初评估 SA 资源时，CPU 增加 ≥8C，Memory 增加 ≥16GB；如果导出文件先存储神策环境，还会占用磁盘资源。【开销取决于：1. 查询语句的复杂度 2. 查询的数据量级 3. 查询导出的频次】

**多产品线注意事项：**
- 如果涉及多产品线，建议 SA 采用集群
- 以上提到的配置 SA 环境中每台数据节点资源都增加，比如客户 SA 是 3+3 标准集群，涉及 SaaS SAT 那么 3 台数据节点每台主机都建议 CPU 增加 2C，Memory 增加 4G，合计至少增加 6C、12GB
- 针对以上这几类会增加资源开销的环境，目前很难给出量化的资源，请提前管理客户预期，避免后续扩容困难

### Phase 4：顺序盘容量计算

顺序盘占用空间的数据来源：
- **event 数据**：有基本评估方法，是计算主体
- **Kafka**：默认持久化 7 天，和导入量相关，不好单独评估
- **user_tag**：根据客户使用情况而定，不能提前评估
- **Yarn**：不能评估
- **ETL 导入数据**：部分客户涉及，需单独评估

因此以下公式**仅计算 event 数据**，最终结果再乘以 1.1 作为整体估算。

**官方计算公式（集群环境）：**

```
event 存储量（GB）= 日导入量（亿）× 35 × 365 / 0.8

含义：
  × 35   → 每亿数据量占用存储空间约 35GB
  × 365  → 计算一年的存储量（按实际保留天数替换）
  / 0.8  → 预留 20% 冗余空间

顺序盘总估算 = event 存储量 × 1.1
             （×1.1 是考虑 Kafka、Yarn、user_tag 等额外占用）

单数据节点顺序盘总容量 = 顺序盘总估算 ÷ 数据节点数
单块顺序盘容量 = 单节点总容量 ÷ 6，向上取整到标准盘型（500G/1T/1.5T/2T）
```

**计算示例（日导入 5000 万，3 个数据节点，保留 1 年）：**
```
event 存储量 = 0.5 × 35 × 365 / 0.8 = 7984 GB
顺序盘总估算 = 7984 × 1.1 = 8782 GB
单节点顺序盘总容量 = 8782 / 3 = 2927 GB
单块顺序盘 = 2927 / 6 ≈ 488 GB → 取整为 500G（或推荐 1000G 留余量）
```

**注意：**
- 保留天数按实际需求替换 365，合规要求可能更长
- 有历史存量时：`(日导入量 × 保留天数 + 历史存量) × 35 / 0.8 × 1.1`
- ETL 导入数据需单独评估后叠加
- 此公式为估算，无法提前精确计算，实际使用中需持续监控磁盘用量

### Phase 5：特殊情况处理

**ARM 架构：**
所有配置容量 double：
- CPU 核数 × 2
- 内存 × 2
- 顺序盘容量 × 2
其他磁盘类型和数量不变。

**IO 密集场景（日导入 + 查询量均较大）：**
- 随机数据盘可适当增加数量以提升 IO 性能
- 日导入超过 3亿/天，建议选择无主机 IO 总限制的云厂商（如腾讯云）
- 避免选择有整机 IO 上限的云厂商（如部分阿里云机型）

**物理机 vs 云主机：**
- 云主机：磁盘可按需扩容，顺序盘可适当保守
- 物理机：不便扩容，配置整体更高，参考 Excel 模板中的物理机配置

### Phase 6：硬件性能标准核查

在输出配置方案前，需确认客户所选硬件满足以下最低性能要求。

#### 磁盘性能要求

| 磁盘类型 | 吞吐标准 | IOPS 标准 | 用途说明 |
|---------|---------|----------|---------|
| 系统盘 | ≥120MB/s | ≥120 | 安装操作系统，存放少量临时日志 |
| 顺序数据盘 | ≥120MB/s | ≥120 | HDFS、Kafka、Yarn、Impala 大数据组件数据 |
| 数据接入暂存盘 | ≥120MB/s | ≥120 | 接收 Nginx 埋点数据 |
| 元数据盘 | ≥150MB/s | ≥5000 | 安装神策组件，存储元数据；需频繁读写 |
| 随机数据盘 | ≥150MB/s | ≥5000 | 存放 Kudu、SKV 数据（用户数据相关）；需频繁读写 |
| 混合数据盘（单机） | ≥150MB/s | ≥5000 | 兼顾顺序+随机角色，存放 Nginx 埋点、HDFS、Kafka、Kudu、SKV 等；需频繁读写 |

**云厂商磁盘类型推荐（满足要求且性价比较高的型号）：**

| 盘用途 | 阿里云 | 腾讯云 | 华为云 | AWS | UCloud | Google Cloud | Azure |
|--------|--------|--------|--------|-----|--------|--------------|-------|
| 系统盘 / 顺序盘 / 暂存盘 | 高效云盘 | 高性能云盘 | 高IO盘 | GP3 | 普通云硬盘 | Balanced persistent disk | 标准HDD |
| 元数据盘 / 随机盘 / 混合盘 | SSD云盘 / ESSD云盘 | 通用型SSD云硬盘 | 通用型SSD | GP3（IOPS>3000 需额外付费） | RSSD云硬盘 | Performance(SSD) persistent disks | 高级SSD |

**重要注意事项：**
- **AWS GP3**：默认 IOPS 为 3000，不满足元数据盘/随机盘/混合盘 ≥5000 的要求，需要额外付费提升 IOPS
- **阿里云**：U1 机型和 R6 机型的 IO 限制不符合标准；R7 及更高机型的随机盘/元数据盘/混合盘只能选 ESSD 云盘（ESSD PL0 满足要求且价格相对较低）
- **当前策略**：可按默认配置购买，后续使用中如监控发现 IOPS 瓶颈，再要求客户付费购买更高 IOPS（需提前告知客户）
- 无论何种磁盘类型，最终均需满足上述吞吐和 IOPS 标准

#### CPU 性能要求

| 最低核数 | events per second（sysbench 素数计算） |
|---------|--------------------------------------|
| ≥8C | ≥14000 |

- **events per second**：每秒处理的事件数量，用 sysbench 工具测试 CPU 每秒的素数计算能力
- CPU 核数 <8C，或 event 计算能力 <14000 时，属于不符合标准，该 CPU 型号性能过低，会影响神策集群服务
- **必须支持的 CPU 指令集**：avx2, avx, ssse3, sse4, sse4_2, popcnt, bmi1, bmi2, pclmulqdq
- 参考链接：https://zhuanlan.zhihu.com/p/642590764

**ARM 架构补充**：ARM CPU 需满足等效计算能力，且所有存储配置 double（见 Phase 5）。

## 常见问题

**客户无法提供准确事件量：**
按 DAU 估算：普通 App DAU×100，电商/金融 DAU×150，得出日均事件量范围。

**客户要求最低配置：**
明确告知风险：Mini 集群最多扩至 5 节点，超出需数据迁移；单机无横向扩容能力。建议客户书面确认。

**物理机配置：**
物理机不便扩容，配置整体比云主机高一档，具体参考 Excel 模板中的物理机配置说明。

**SF 配置：**
SF 独立于 CDP 部署，配置表待补充，暂时告知客户需单独评估。
