# /sd-design-test — 测试方案设计

> ⚠️ **执行前确认**
>
> **此 command 会做什么：**
> 根据当前项目阶段，设计对应的测试方案（SIT Plan、UAT Plan 或 Perf Plan）。
>
> **前置条件：**
> - 项目已初始化（PROJECT.md 存在）
> - 了解当前系统状态（开发完成度、数据准备情况）
>
> **执行步骤概览：**
> 1. 确认测试类型（SIT / UAT / Perf）
> 2. 收集测试需求（范围、标准、环境）
003e 3. 设计测试方案（用例、数据、工具）
> 4. 输出测试计划文档
>
> 1/y = 确认执行
> 0/n = 取消
> 2/s = 跳过

## 工作流

### Step 1：选择测试类型

根据项目当前阶段选择：

| 测试类型 | 适用阶段 | 目的 |
|----------|----------|------|
| **SIT** | 系统开发完成 | 验证模块间集成和数据流转 |
| **UAT** | SIT 通过 | 用户验收，验证业务场景 |
| **Perf** | UAT 通过 | 验证系统性能和容量 |

### Step 2：设计 SIT Plan

**适用条件**：系统已部署，数据管道已完成

**内容**：
1. 测试范围确认（业务场景列表、验收标准）
2. 环境信息（URL、测试账号、截止时间）
3. 测试用例设计（TC-001 ~ TC-XXX）
4. 数据准备要求
5. 通过标准（P0/P1/P2 分级）

**输出**：`test-plans/sit-plan.md`

### Step 3：设计 UAT Plan

**适用条件**：SIT 已通过，看板已创建

**内容**：
1. 验收范围（与埋点方案对齐）
2. 测试账号准备（UAT-001 ~ UAT-N）
3. 验收脚本设计（按业务场景拆解）
4. 缺陷分级标准（致命/严重/一般/轻微）
5. 通过标准（所有 P0 通过）

**输出**：`test-plans/uat-plan.md`

### Step 4：设计 Perf Plan

**适用条件**：UAT 已通过，待测试环境已部署

**内容**：
1. 性能目标（并发用户数、P95 响应时间、QPS/TPS）
2. 测试场景（事件上报、分群计算、看板查询）
3. 测试数据量级
4. 测试工具和方法（JMeter/LoadRunner/自定义脚本）
5. 通过标准（无内存泄漏、CPU < 80%、响应时间达标）

**输出**：`test-plans/perf-plan.md`

## 完成建议

- "UAT Plan 设计完成？→ 下一步 `/sd-test-run` 执行 UAT"
