# /sd-run-test — 测试执行

> ⚠️ **执行前确认**
>
> **此 command 会做什么：**
> 根据已设计的测试方案，执行对应的测试（SIT / UAT / Perf）。
>
> **前置条件：**
> - 对应测试方案已设计完成（`/sd-test-design`）
> - 测试环境已就绪
>
> **执行步骤概览：**
> 1. 选择测试类型（SIT / UAT / Perf）
> 2. 确认测试方案和环境
> 3. 执行测试（逐条验证/压测）
> 4. 记录结果和缺陷
> 5. 输出测试报告
>
> 1/y = 确认执行
> 0/n = 取消
> 2/s = 跳过

## 工作流

### Step 1：选择测试类型

| 测试类型 | 前置条件 | 输出 |
|----------|----------|------|
| **SIT** | 系统部署完成，数据管道就绪 | SIT 测试报告 |
| **UAT** | SIT 通过，看板已创建 | UAT 验收报告 |
| **Perf** | UAT 通过，环境已部署 | 性能测试报告 |

### Step 2：执行 SIT

**输入**：`test-plans/sit-plan.md`

**流程**：
1. 确认测试环境（URL、账号可用性）
2. 按用例逐条执行（TC-001 ~ TC-XXX）
3. 记录实际结果，标记状态（通过/失败/阻塞）
4. 缺陷记录和分类

**输出**：
```
## SIT 测试报告

**汇总**
- 总用例数：X
- 通过：X（X%）
- 失败：X
- 阻塞：X

**上线建议**
[ ] 可以上线（所有 P0 通过）
[ ] 不建议上线（存在 P0 失败）
```

### Step 3：执行 UAT

**输入**：`test-plans/uat-plan.md`

**流程**：
1. 确认验收范围与埋点方案一致
2. 确认测试账号可用
3. 按验收脚本逐条验证
4. 缺陷跟踪（记录、分类、修复验证）

**输出**：
```
## UAT 验收报告

**汇总**
- 验收项：X
- 通过：X
- 缺陷：X（致命/严重/一般/轻微）

**上线建议**
[ ] 可以上线
[ ] 不建议上线
```

### Step 4：执行 Perf

**输入**：`test-plans/perf-plan.md`

**流程**：
1. 基准测试（单场景，建立基线）
2. 负载测试（逐步加压，找到拐点）
3. 压力测试（极限容量，验证稳定性）
4. 稳定性测试（长时间运行，检查内存泄漏）

**输出**：
```
## 性能测试报告

**测试环境**
- 服务器配置：...
- 数据量级：...

**测试结果**
| 场景 | 并发 | P50 | P95 | P99 | QPS | 错误率 |
|------|------|-----|-----|-----|-----|--------|
| ... | ... | ... | ... | ... | ... | ... |

**瓶颈分析**
- ...

**优化建议**
- ...
```

## 完成建议

- "SIT 未通过？→ 修复缺陷后重新执行 `/sd-test-run`"
- "SIT 通过？→ 设计 UAT Plan：`/sd-test-design`"
- "UAT 通过？→ 设计 Perf Plan：`/sd-test-design`"
