## Topic: 服务器容量评估

> 状态：✅ 可用（执行入口）
> 适用场景：新客户部署前、现有客户扩容评估
> 详细计算：请调用 `/sd-server-sizing`

**快速决策：**

| 日事件量 | 部署模式 | 参考配置 |
|---------|---------|---------|
| ≤1000万/天 | 单机标配 | 8C / 64G |
| ≤1500万/天 | 单机高配 | 16C / 128G |
| ≤6500万/天 | Mini 集群标配 | (8C/64G) ×3 |
| ≤1.5亿/天 | Mini 集群高配 | (16C/128G) ×3 |
| >1.5亿/天 | 标准集群 | 3元数据+N数据节点，需运行 sizing_calc.py |

**关键陷阱：**
- Mini 集群最多扩至 5 节点，超量需迁移至标准集群（有数据迁移成本）
- ARM 架构所有配置 double（CPU/内存/顺序盘均 ×2）
- 顺序盘容量必须用脚本计算，禁止人工估算：`sizing_calc.py --daily-events <N>`
- AWS GP3 默认 IOPS 3000，不满足元数据盘 ≥5000 要求，需额外付费提升

**评估问卷（向客户收集）：**
日均事件量、DAU、历史存量事件量、数据保留天数、未来12个月增长预期、CPU架构(X86/ARM)、是否需要SF/Abtest等附加产品

### 常见问题（来自 `/sd-server-sizing` 实战沉淀）

**客户无法提供准确事件量：**
按 DAU 估算：普通 App DAU × 50～100 事件/天，电商/金融类 App DAU × 100～200 事件/天。

**客户要求最低配置：**
明确告知风险：Mini 集群最多扩至 5 节点，超出需数据迁移；单机无横向扩容能力。建议客户书面确认。

**物理机配置：**
物理机不便扩容，配置整体比云主机高一档，具体参考 Excel 模板中的物理机配置说明。

**SF（SFN/MA）配置：**
SF 独立于 CDP 部署，按 `/sd-server-sizing` Phase 8 的集群档位表选型。日活 ≥2000万 或 事件量 ≥10亿 时，需联系 SaaS 运维单独评估。使用新资源位、OpenAPI、实时受众、业务事件等功能时，需额外核查业务量限制。

## Topic: 网络带宽评估

> 状态：✅ 可用（执行入口）
> 适用场景：数据接入方案设计、集成通道规划
> 详细计算：请调用 `/sd-server-sizing` Phase 1

**带宽计算公式：**
```
所需带宽 = QPS × 平均包大小 × 冗余系数(1.2~1.5)
```

**接入方式对比：**

| 方式 | 带宽消耗 | 适用场景 |
|------|---------|---------|
| SDK | 低（批量+压缩） | App/Web 前端埋点 |
| HTTP API | 中（逐条/小批量） | 服务端事件、第三方系统 |
| SFTP | 高（全量文件传输） | 离线数据导入 |

**内网 vs 公网：**
- 公网接入：建议 HTTPS + 考虑峰值冗余系数 1.5
- 内网接入：冗余系数 1.2，注意 VPC 内部无额外加密（标准方案）

**典型案例：**
- DAU 100万 × 100事件/天 ≈ 1000万事件/天，SDK 批量上报带宽约 5~10Mbps

## Topic: 信创交付

> 状态：✅ 可用（与 NFR 合并）
> 适用场景：客户要求国产化、政府/金融行业项目
> 关联 skill：`/sd-tech-design`（LLD Section 10.6 Compliance）、`/sd-server-sizing`（ARM 配置 double）

### 统一信创矩阵

| 维度 | 类别 | 支持项 | 处理方式 | 备注 |
|------|------|--------|----------|------|
| **国产 OS** | 操作系统 | 麒麟、统信 UOS | 推荐 Rocky Linux 9.6 作为替代（通用性更好） | 神策 CDP/MAE 基于 Linux 内核，Rocky 兼容性已验证 |
| **国产 CPU** | 处理器 | 鲲鹏、飞腾(ARM) | 支持，所有配置 double（CPU/内存/顺序盘） | 需运行 `sizing_calc.py --arch ARM` |
| **国产数据库** | 数据库 | 达梦、人大金仓 | 神策使用 RDS 托管 MySQL，数据库由云厂商/客户负责 | 不直接依赖特定国产数据库 |
| **国产中间件** | 消息队列 | 无（Kafka 为必需） | 不支持替换 | Kafka 为神策核心组件，暂无国产替代方案 |
| **国产云** | 云平台 | 阿里云、华为云、腾讯云 | 支持，按标准部署流程执行 | 华为云需注意 ARM 实例选型 |
| **信创合规** | 法规 | 等保 2.0、GB/T 20988-2007 | LLD Section 10.6 输出合规对照表 | Tech Spec 补充具体实施细节 |

### 不支持项处理方式

| 不支持项 | 原因 | 替代方案 | 需客户确认 |
|----------|------|----------|-----------|
| Windows Server 部署 | 神策仅支持 Linux | Rocky Linux 9.6 | 是 |
| 国产数据库直接替换 MySQL | 神策依赖 RDS 托管 MySQL | 由云厂商/客户提供兼容 RDS | 否（云厂商责任） |
| Kafka 替换为国产消息队列 | 核心组件深度绑定 | 无替代方案 | 是（如强制要求则项目不可行） |
| 非 ARM 架构国产 CPU | 未验证 | 仅支持鲲鹏/飞腾(ARM) | 是 |

### 标准回复框架

1. 神策 CDP/MAE 部署在 Linux 环境，推荐 Rocky Linux 9.6
2. 数据库层使用 RDS 托管，不直接依赖特定国产数据库
3. ARM 架构支持，但需 double 资源配置（CPU/内存/顺序盘）
4. 具体兼容性问题请联系 SaaS 运维单独评估

## Topic: 私有化部署 NFR 快速决策表

> 状态：✅ 可用
> 适用场景：售前技术评估、LLD Section 6/7/8 填写、客户技术答疑
> 关联 skill：`/sd-tech-design`（LLD 完整章节）、`/sd-server-sizing`（容量与性能）

### 12 维度 NFR 快速决策表

| 维度 | 关键问题 | 标准答案 / 决策规则 | LLD 章节 |
|------|----------|---------------------|----------|
| **1. 可用性** | 系统可用性目标？ | 标准集群 ≥99.9%，Mini 集群 ≥99.5% | Section 7.1 |
| **2. RTO** | 故障恢复时间目标？ | 元数据节点 ≤4h，数据节点 ≤8h | Section 7.1 |
| **3. RPO** | 数据丢失容忍度？ | 元数据 ≤15min，数据节点 ≤1h | Section 7.1 |
| **4. 扩容性** | 未来扩容路径？ | Mini 集群最多 5 节点，超量需迁移标准集群 | Section 6.2 |
| **5. 性能** | 查询响应时间要求？ | 简单查询 ≤3s，复杂查询 ≤10s，报表 ≤30s | Section 5.3 |
| **6. 监控** | 监控工具要求？ | CloudMonitor + Prometheus 混合方案 | Section 8.1 |
| **7. 备份** | 备份频率与保留期？ | 每日增量 + 每周全量，保留 30 天 | Section 8.2 |
| **8. 安全** | 传输/静态加密？ | TLS 1.2+ / AES-256 + KMS | Section 10.2 |
| **9. 合规** | 等保/信创要求？ | 等保 2.0 三级：参考 LLD Section 10.6 对照表 | Section 10.6 |
| **10. 维护窗口** | 允许停机时间？ | 每周 ≤4h，需提前 48h 通知 | Section 8.3 |
| **11. 网络隔离** | VPC/专线要求？ | 标准方案：VPC 内网 + VPN 跳板机 | Section 3.5.1 |
| **12. 数据保留** | 在线保留期？ | 默认 1 年，超期归档至 OSS/COS | Section 8.2 / Tech Spec |

### NFR 与信创的关系

- **NFR 覆盖**：通用技术约束（可用性、性能、安全、运维等 12 维度）
- **信创覆盖**：国产化特定问题（OS/CPU/DB/合规）
- **交集**：等保合规（NFR 维度 9 与信创矩阵最后一行）
- **处理原则**：先过 NFR 快速决策表确认通用约束，再查信创矩阵确认国产化兼容性

## Topic: ID3 设计规范

> 状态：✅ 可用（执行入口）
> 适用场景：技术方案评审、架构图交付
> 详细绘制：请调用 `/sd-draw-diagram`

**必要要素：**
- 系统边界（CDP / MAE / 外部集成系统）
- 数据流向（带方向箭头，标注协议）
- 集成点（API / SFTP / SDK / Kafka）
- 环境分区（PRD / DR / UAT / DEV）

**标注规范：**
- 命名：组件名 + 版本号（如 SA 3.7.2）
- 图例：统一放在右下角
- 版本号：文档名中包含日期（如 `Logical_Architecture_20260414.drawio`）

**常见错误：**
- ❌ 缺少数据流方向
- ❌ 边界不清晰（CDP 与外部系统混为一谈）
- ❌ 未标注环境（PRD/UAT 画在同一张图）

### 常见问题（来自 `/sd-draw-diagram` 实战沉淀）

**arch.yaml 中 future 节点是否需要连线？**
可以有 future 连线（`status: future`），渲染为灰色虚线，表示规划中的数据流。

**字段名应该用中文还是英文？**
fields 列表用英文 snake_case（渲染成连线标签的一部分）；name 用英文短语（客户审阅用）。

**需要在 PPT 中使用：**
draw.io → File → Export As → PNG（300dpi）或 SVG → 嵌入 PPT。

**交付物检查清单：**
- [ ] 逻辑架构图（Section 3.3）
- [ ] 系统流图（End User / Business User / Maintenance User）
- [ ] 基础设施架构图（VPC/子网/安全组件）
- [ ] 数据流图（CDP + MAE 分别绘制）
- [ ] 功能架构图（Appendix）

### 常见问题（来自 `/sd-tech-design` 实战沉淀）

**客户不提供节点 IP 等具体信息：**
LLD Section 5.1 填节点数量，IP 留 `<TBD>`，在 Section 11.2 Pre-requisites 中列为前置条件。

**客户要求简化 LLD：**
不可省略的核心章节：Section 1/2/3/5/6/7/10/11。可简化：Section 4（选型简单时）、Section 8（运维由厂商负责时）。

**客户要求用 PPT 内嵌图而非独立 draw.io 文件：**
先用 `/sd-draw-diagram` 生成 draw.io 文件，客户调整后导出 PNG/SVG 嵌入 PPT。

**Tech Spec 和 LLD 内容重复时：**
Tech Spec 用"参见 LLD Section X.X"引用，不重复写正文，只补充 LLD 没有的实施细节。

**客户对某节 LLD 反复修改：**
每次修改在文档版本历史中记录（修改内容 + 修改原因 + 审批人），这是合规审计的证据链。

## Topic: 问题排查 SOP

> 状态：✅ 可用
> 适用场景：客户反馈异常、上线后数据问题

**症状 → 检查步骤 → 关联 Skill：**

| 症状 | 第一步检查 | 第二步检查 | 关联 Skill |
|------|-----------|-----------|-----------|
| 数据延迟高 | 埋点方案版本 vs 代码版本是否一致 | 事件触发时机是否正确 | `/sd-data-validation` Phase 3 |
| 事件不入库 | 测试设备是否在神策白名单内 | SDK 初始化是否成功 | `/sd-tracking-setup-e2e` Step 1b |
| 属性值为 null | 区分"代码未传值"vs"业务场景确实无值" | 更新方案说明或修复代码 | `/sd-data-validation` Phase 3 |
| 数据丢失/重复 | 检查导入任务的幂等性配置 | 检查 Kafka 消费偏移量 | `/sd-tracking-setup-e2e` Phase 4 |
| 分群计算异常 | 检查用户属性是否最新 | 检查分群条件逻辑 | `/sd-data-validation` Phase 3 |

**升级路径（何时上报）：**
- 排查 30 分钟无结论 → 记录到 `/sd-feedback` 并通知技术经理
- 涉及数据丢失且无法恢复 → 立即上报，启动 DR 流程
- 客户要求书面报告 → 使用 `/sd-data-validation` 输出差异报告

**已知陷阱（来自各 Skill 实战）：**
- ARM 架构配置未 double 导致性能不足 → `/sd-server-sizing` Phase 5
- AWS GP3 IOPS 不足导致元数据盘性能瓶颈 → `/sd-server-sizing` Phase 6
- 埋点方案未确认就造数 → `/sd-tracking-setup-e2e` Iron Law
- YAML 枚举值未确认就导入 → `/sd-tracking-setup-e2e` Step 1c
- 跳过用例设计直接测试 → `/sd-sit-uat` Iron Law
- UAT 由交付团队代劳 → `/sd-uat` Iron Law
- 仅凭印象判断数据正确 → `/sd-data-validation` Iron Law
- 性能测试先于容量规划 → `/sd-performance-test` 与 `/sd-server-sizing` 执行顺序

### 常见问题（来自各 Skill 实战沉淀）

**来自 `/sd-tracking-setup-e2e`：**
- **客户说"先做再确认"：** 坚持方案确认门。方案变更后造数和看板全部作废，返工成本远高于等待确认的时间。
- **import_meta_data.py 报"缺少必要配置"：** 在客户项目目录下运行 `sdeliver check`，确认 `.env` 中必填项已填写。
- **字段创建报 ALREADY_EXISTS：** 正常，脚本自动跳过，不影响其他字段。
- **validate_import.py 自定义查询无数据：** 数据处理有延迟，加 `--wait 60` 等待后重试。若仍无数据，检查账号是否有自定义查询权限。
- **模拟数据不够真实：** 向客户要 1-2 个真实业务场景的描述，更新 `rules/business_logic.yaml` 中的 `event_sequences`，重新验证后造数。

**来自 `/sd-sit-uat`：**
- **客户要求跳过用例设计直接测试：** 解释没有用例无法出具正式测试报告，且无法证明覆盖了哪些场景。最少也要有一个场景清单。
- **P0 用例失败但客户坚持上线：** 书面记录风险，要求客户签字确认，在报告中注明"客户知情并接受风险"。

**来自 `/sd-uat`：**
- **客户没有神策后台账号：** 联系项目经理申请只读权限账号，UAT 期间临时开放。
- **Phase 4d 报告有大量 warning：** 在 UAT 场景设计中优先覆盖 warning 项对应的事件/属性，确认客户是否接受。
- **客户认为数据量太少：** 解释 UAT 使用的是模拟数据，生产环境数据量会更大。如客户坚持，可协调 tracking-setup-e2e 重新造数（增加 `--users` 和 `--days`）。

**来自 `/sd-data-validation`：**
- **找不到埋点方案文档：** 使用 `/sd-tracking-setup-e2e` 补充方案后再校验。
- **数据量为零：** 先确认测试设备是否在神策的设备白名单内，再确认 SDK 初始化是否成功。
- **属性值为 null：** 区分"代码未传值"和"业务场景确实无值"，前者是 bug，后者需要更新方案说明。

**来自 `/sd-performance-test`：**
- **测试环境未部署就要求压测：** 必须先完成 `/sd-server-sizing` 容量规划并部署完成，才能执行性能测试。无环境则无基线可测。
- **客户要求压测生产环境：** 明确拒绝。性能测试必须在 UAT/压测环境执行，禁止直接压生产。

## 知识库结构化原则

1. **FAQ 是导航层，不是内容层**：详细执行流程保留在各业务 Skill 中，FAQ 只提供快速决策表、陷阱清单和关联入口
2. **症状导向**：排查类问题按"症状 → 检查步骤 → 关联 Skill"结构组织，不重复写执行细节
3. **版本标注**：每个 Topic 标注状态（🚧/✅）和最后更新日期，来源标注在内容中
4. **反馈联动**：FAQ 底部「已知陷阱」小节引用各 Skill 的常见问题，保持单向同步（Skill → FAQ）

## FAQ 内容工程规范（维护者必读）

本 skill 定位为「知识导航层」，不重复存放各业务 skill 的执行细节。新增或修改 Topic 时遵循以下格式：

### 关键原则

1. **导航层不替代执行层**：FAQ 只给快速决策表、陷阱清单、关联入口。详细 Phase/Step 保留在业务 skill 中。
2. **必须标注关联 skill**：每个 Topic 头部用 `关联 skill` 标明执行入口，方便用户跳转。
3. **症状导向的排查 SOP**：排查类问题按「症状 → 检查步骤 → 关联 skill Phase/Step → 升级路径」结构组织。
4. **版本同步**：FAQ 引用 skill 的具体 Phase/Step 时，需与该 skill 当前版本保持一致。skill 更新后，FAQ 需同步检查引用是否失效。
5. **状态标注**：🚧 表示仅有提纲或待补充；✅ 表示导航层内容完整（不要求穷尽所有细节）。
