# 埋点全链路交付

## 适用场景

- 客户新项目启动，需要从零建立数据采集体系
- 现有业务新增功能，需要补充埋点和分析看板
- 需要将测试环境配置迁移到客户生产环境

## 核心原则（Iron Law）

**必须先完成采集方案设计并经客户确认，再进行后续任何步骤。**

不允许在方案未确认的情况下造数或创建看板。违反此原则会导致返工。

## 执行阶段

### Phase 1：业务目标确认

收集以下信息后才能进入 Phase 2：
- 核心业务场景（用户注册/购买/使用路径等）
- 需要回答的关键业务问题（转化率？留存？活跃度？）
- 分析维度（渠道、用户分群、时间粒度）
- 数据消费方（运营/产品/数据分析师）

输出：业务目标确认单（一页，列出以上四项）

### Phase 2：采集方案设计

基于 Phase 1 输出，设计埋点方案：
- 事件列表（事件名、中文名、触发时机）
- 每个事件的属性列表（属性名、类型、示例值、是否必填）
- 全局属性（用户 ID、设备信息、版本号等）

**等待客户确认后才能进入 Phase 3。**

输出：埋点方案表（Markdown 表格或 Excel）

### Phase 3：模拟数据生成

使用 `generate_mock_data.py` 脚本从埋点方案 Excel 自动生成模拟数据：

```bash
# 安装依赖（首次）
python3 -m venv .venv && source .venv/bin/activate
pip install openpyxl

# 生成模拟数据（每个事件 50 条，20 个模拟用户）
python3 tracking-setup-e2e/generate_mock_data.py \
  --input "埋点采集方案.xlsx" \
  --output ./mock_data \
  --count 50 \
  --users 20
```

输出：
- `mock_data/mock_events.jsonl` — 每行一条记录，可逐行检查
- `mock_data/mock_events_batch.txt` — base64 编码，可直接 POST 到神策 HTTP API

通过 HTTP API 导入：
```bash
curl -X POST 'https://<sa-host>/sa?token=<token>' \
     -d 'data='$(cat mock_data/mock_events_batch.txt)
```

### Phase 4：环境导入与看板创建

1. 将模拟数据导入目标测试环境
2. 验证数据已正确入库（抽查 3-5 个事件）
3. 按业务目标创建看板（参考 cdp-operations skill）
4. 截图存档，输出看板创建 SOP

> 当前阶段神策系统无 API，看板创建通过 browser automation 或手动操作完成。

输出：导入确认截图 + 看板截图 + 操作 SOP

### Phase 5：资产迁移

使用神策资产项工具将配置元数据从测试环境迁移到客户生产环境：
1. 导出测试环境资产（看板、标签、事件定义）
2. 检查资产依赖关系
3. 导入客户生产环境
4. 验证迁移结果

输出：资产迁移记录（迁移项清单 + 验证截图）

## 输出模板

```
## 交付物清单

- [ ] 业务目标确认单
- [ ] 埋点方案表（已客户确认）
- [ ] 模拟数据文件
- [ ] 环境导入确认截图
- [ ] 看板截图及操作 SOP
- [ ] 资产迁移记录
```

## 常见问题

**客户说"先做再确认"：** 坚持 Phase 2 确认门。方案变更后造数和看板全部作废，返工成本远高于等待确认的时间。

**模拟数据不够真实：** 向客户要 1-2 个真实业务场景的描述，按真实路径生成数据。

**资产迁移失败：** 先检查目标环境是否有同名资产冲突，再检查数据源依赖是否满足。
